================================================================================
KEEL3 WEEKLY INTELLIGENCE BRIEF
FULL SYSTEM REBUILD GUIDE
================================================================================

PURPOSE
This document describes how to build the Keel3 "Heads Down to Heads Up"
weekly intelligence brief system from scratch, in a new environment, with no
existing infrastructure. It produces the complete set of accounts, databases,
scheduled tasks, skills, and prompts required to run the system. Once setup
is complete, switch to the companion document "Weekly Assembly Runbook" for
the recurring operation.

AUDIENCE
A developer or operator setting up the system in a new Claude Code
environment. Assumes basic familiarity with Python, Notion, Gmail, and
Anthropic's Claude products.

ESTIMATED TIME
Roughly one full work day for an experienced operator. Two days if Notion
schema, OAuth flows, or the renderer Python environment need debugging.

AUTOMATION LEGEND
Each step is labeled with one of three tags:

  [AUTOMATED]    The step runs without human intervention once configured.
  [HUMAN REQUIRED] A person must perform this step.
  [AUTOMATABLE]  Currently human; could be automated with the work noted.

PHASE A: PREREQUISITES AND ACCOUNTS
================================================================================

--------------------------------------------------------------------------------
STEP 1: ANTHROPIC CLAUDE ACCOUNT WITH SCHEDULED TASKS ACCESS
--------------------------------------------------------------------------------
[HUMAN REQUIRED]

Sign up at claude.ai. Subscribe to a plan that includes:
- Claude Code access (terminal-based session)
- Scheduled Tasks (cloud-side recurring task runner)
- User skill installation

At the time of writing, Claude Pro or Claude Max plans include these. Verify
in account settings that "Scheduled Tasks" is visible in the left-hand
sidebar of claude.ai. If not visible, upgrade the plan.

--------------------------------------------------------------------------------
STEP 2: GROK ACCOUNT WITH SCHEDULED TASKS ACCESS
--------------------------------------------------------------------------------
[HUMAN REQUIRED]

Sign up at grok.com (X / Twitter Premium login). Verify access to
grok.com/tasks. This is the scheduling interface for recurring Grok queries.
At the time of writing, this feature is available on the higher X Premium
tiers (Premium+ or SuperGrok). Verify by visiting grok.com/tasks; if the
page loads with a "Create Task" button, access is good.

--------------------------------------------------------------------------------
STEP 3: NOTION WORKSPACE
--------------------------------------------------------------------------------
[HUMAN REQUIRED]

Create or use an existing Notion workspace. The workspace must be paid
(Plus tier or above) to support the database row count expected (50 to 200
new rows per month across all weeks). Verify that the workspace allows API
integrations and OAuth connections.

If using an existing workspace, create a top-level page called "Keel3" to
serve as the parent for the brief infrastructure. Note its URL for later.

--------------------------------------------------------------------------------
STEP 4: GMAIL OR EQUIVALENT EMAIL
--------------------------------------------------------------------------------
[HUMAN REQUIRED]

Use a Gmail account (personal or Google Workspace) for two purposes:
(a) receiving Grok scheduled task emails, and
(b) sending the final client briefs.

If newsletter-style AI emails are intended to also feed the Daily Scan,
configure a Gmail label called "AI" or "AI Newsletters" and apply it to
relevant subscriptions (The Information, Stratechery, Ben's Bites, etc.).

OAuth between Gmail and Claude scheduled tasks will be set up later in
Step 14.

PHASE B: NOTION STORY POOL CONSTRUCTION
================================================================================

--------------------------------------------------------------------------------
STEP 5: CREATE THE STORY POOL DATABASE
--------------------------------------------------------------------------------
[HUMAN REQUIRED, with Claude assist]

Inside the Notion "Keel3" page, create a new full-page database. Name it:

    AI Brief Story Pool

The database holds one row per AI news story being tracked. Schema is fixed
and detailed in Appendix A. The fastest way to create it correctly is to
have Claude do it via the Notion API. From a Claude Code session connected
to Notion via OAuth (see Step 14 for OAuth setup; you may want to swap
Step 5 and Step 14 if not already connected), prompt:

    "Inside the Keel3 page at [your Notion page URL], create a database
     called 'AI Brief Story Pool' with the schema specified in Appendix A
     of the Keel3 Full System Rebuild Guide. Use the exact property names,
     types, and select options listed there."

Verify the database appears in Notion with all 12 properties.

[AUTOMATABLE] Already largely automated through Claude's Notion create-
database tool. The human step is providing the parent page URL and
confirming the result.

--------------------------------------------------------------------------------
STEP 6: ADD SELECT OPTIONS TO MULTI-SELECT AND SELECT PROPERTIES
--------------------------------------------------------------------------------
[HUMAN REQUIRED, with Claude assist]

Several properties (Resonance, Relevance, Lens, Status, Capture Source,
Week) are select or multi-select fields with predefined options. The exact
option lists and recommended colors are in Appendix A.

If Claude created the database in Step 5, the options should already exist.
If not, manually add them in Notion's database settings, or prompt Claude:

    "On the AI Brief Story Pool database, add the select options for
     Relevance, Lens, Status, Capture Source, Resonance, and Week as
     specified in Appendix A."

Notion API does not allow programmatic recoloring of select options after
creation. If the colors do not match the brand palette, manually adjust
them in the Notion UI by clicking each select option's three-dot menu
and choosing a color. The recommended mapping:

  Capture Source - Daily Scan: Blue
  Capture Source - Grok Email: Orange (must be set manually; default may
                                 be a different color)
  Capture Source - Newsletter: Green
  Capture Source - Manual: Gray

--------------------------------------------------------------------------------
STEP 7: CREATE THE EIGHT VIEWS ON THE DATABASE
--------------------------------------------------------------------------------
[HUMAN REQUIRED, with Claude assist]

The database needs eight views, each with specific filter and sort rules:

  1. Default (table view, no filter, sort by Date Added desc)
  2. This Week (filter Date Added in last 7 days, sort by Resonance)
  3. By Week (grouped by Week field, manual sort within groups)
  4. KPS Life (filter Relevance contains KPS Life AND Status != Archived,
              sort by Resonance)
  5. M&T Bank (filter Relevance contains M&T Bank AND Status != Archived,
              sort by Resonance)
  6. Brown Advisory (same pattern for Brown Advisory)
  7. Alpine Associates (same pattern)
  8. Incserv (same pattern)
  9. ChristianaCare (same pattern)
 10. Luna Diabetes (same pattern)

(That is actually ten views once the per-client views are listed individually
but the core structural views are eight: Default, This Week, By Week, and
the seven client views.)

Have Claude create them:

    "On the AI Brief Story Pool database, create the views specified in
     Appendix A of the Full System Rebuild Guide. Apply the exact filter
     rules and sort orders listed."

Verify each view in the Notion UI by opening it and confirming the filter
is correct.

--------------------------------------------------------------------------------
STEP 8: RETRIEVE AND RECORD THE DATABASE ID AND DATA SOURCE ID
--------------------------------------------------------------------------------
[HUMAN REQUIRED]

The database has two key identifiers:

  - Database ID: the page-level identifier visible in the URL when the
    database is open as a full page. Format:
    https://www.notion.so/<workspace>/<DATABASE_ID>
    The DATABASE_ID is the 32-character hex string.

  - Data Source ID: the underlying data source identifier, formatted as a
    UUID with hyphens. This is needed for the Notion API search tool.
    Obtain it via: ask Claude to call notion-fetch on the database URL.
    Claude will return both IDs in the schema response.

Record both IDs in a secure note. They will be referenced in:
- The Daily Scan task prompt (Step 16)
- The skill's build_template.py (if hardcoding rather than searching)
- Any direct API tooling built around the database

For the reference deployment, these were:

  Database ID:    c02c8d084c4348e09483b0f31e5606ef
  Data Source ID: 2127ca62-01e2-432e-af43-c19579a8d014

These will be different in any new deployment.

--------------------------------------------------------------------------------
STEP 9: CREATE A NOTION INTEGRATION FOR API ACCESS
--------------------------------------------------------------------------------
[HUMAN REQUIRED]

Visit www.notion.so/profile/integrations and create a new internal
integration named "Keel3 Brief Pipeline." Grant it read and write access
to the workspace. Copy the integration's secret token. Then go to the
Keel3 parent page in Notion and add the integration via Settings > Connections.

This token will be referenced by the Claude scheduled task in Step 14 if
the task is configured to talk directly to Notion via API key rather than
via the OAuth MCP connector. In most cases the OAuth MCP connector is
preferable; the API key approach is a fallback.

PHASE C: GROK SCHEDULED TASK
================================================================================

--------------------------------------------------------------------------------
STEP 10: CREATE THE GROK SCHEDULED TASK
--------------------------------------------------------------------------------
[HUMAN REQUIRED]

Visit grok.com/tasks. Click "Create Task." Configure:

  Name:        Keel3 Daily AI Scan
  Frequency:  Daily
  Time:       6:00 a.m. ET (or preferred morning slot, before Daily Scan)
  Output:     Email to <operator email>

Paste the prompt from Appendix B into the task body. The prompt covers six
curated X lists chosen specifically for high-signal AI coverage:
- AI Newsmakers
- AI Founders
- AI Investors
- AI Policy
- Healthcare/Science
- AI Researchers/Developers

The prompt also includes flagging rules (de-emphasize crypto and token
framing, flag single-post claims for verification, deliver full output
not truncated digest).

Save the task.

--------------------------------------------------------------------------------
STEP 11: VERIFY GROK EMAIL DELIVERY
--------------------------------------------------------------------------------
[HUMAN REQUIRED]

Either wait for the next scheduled run or trigger the task manually via
the "Run Now" button. Check the operator's email inbox for the delivered
scan. Verify:

  - Subject line is clear
  - Body contains the structured output with story titles, sources,
    summaries, and a "noise" section at the bottom
  - No truncation midway through the output
  - Sources include X post URLs where applicable

If the delivery fails or the output is truncated, common fixes:
- Check Grok task configuration for "deliver full output" toggle
- Verify the email address in the task settings matches the operator's
  inbox
- Try a smaller scan scope first (fewer lists) and add back gradually

--------------------------------------------------------------------------------
STEP 12: SET UP A GMAIL LABEL OR FILTER FOR THE GROK EMAILS
--------------------------------------------------------------------------------
[HUMAN REQUIRED]

In Gmail, create a label "Grok Scan" and a filter that automatically
applies the label to emails from the Grok scheduled task. This makes the
emails easy to find each morning and easy to script against later.

[AUTOMATABLE] When the Claude Daily Scan task is configured in Phase D,
have it read directly from this Gmail label rather than requiring the
human to copy-paste each morning. This is the Step 2 automation noted in
the Weekly Assembly Runbook.

  Name:       Keel3 Daily AI News Scan
  Frequency: Daily

              available)
  Tools:     Notion (write), Gmail (read), Web Search

--------------------------------------------------------------------------------
STEP 16: PASTE THE STANDING DAILY SCAN PROMPT
--------------------------------------------------------------------------------
[HUMAN REQUIRED]

Paste the prompt from Appendix C into the task body. This is the standing
instruction set that the scheduled task runs each morning. It tells Claude:

  - The seven client profiles and lenses to track
  - How to read newsletter emails and web-search trending AI news
  - How to triage stories into Notion rows
  - How to write the shared layer for each row
  - How to tag each row with Relevance, Lens, Resonance, Capture Source
    (Daily Scan), and Week
  - How to check for existing similar rows before creating duplicates
  - How to flag stories that need human verification

The prompt also includes the Notion database ID and data source ID from
Step 8.

Save the task.

[AUTOMATABLE NOTE] Two improvements to the standing prompt that close
known gaps in early deployments:
  - Add: "Check the Federal Register and the news pages of FDA.gov,
    OCC.gov, federalreserve.gov, FDIC.gov, CISA.gov, and SEC.gov each
    day for new notices in the prior 24 hours." This catches the kind of
    procedural regulatory updates that do not travel through X.
  - Add: "When filing a new row, first search the Story Pool for similar
    titles. If a row exists for the same announcement, update or skip
    rather than creating a duplicate."
These are already in the Appendix C version of the prompt.

STEP 18: INSTALL THE SKILL
--------------------------------------------------------------------------------
[HUMAN REQUIRED]

Install user skills through Anthropic's user-skills mechanism. As of the

  Path A: claude.ai Settings > Skills > Upload Skill. Drag the zip file.

  Path B: For Claude Code environments where user skills are filesystem-

          /mnt/skills/user/keel3-weekly-brief-pipeline/SKILL.md

          /mnt/skills/user/keel3-weekly-brief-pipeline/render_brief.py

If installation fails with "SKILL.md must start with YAML frontmatter (---)"
or a similar error, verify the SKILL.md file begins with the three dashes
on line 1 and contains a properly closed frontmatter block before any other
content.

STEP 20: TEST RENDER WITH THE EXAMPLE BRIEF

APPENDIX A: NOTION DATABASE SCHEMA
================================================================================

Database name: AI Brief Story Pool

PROPERTIES (12 total)

  1. Story Title
     Type: title (required, primary key)
     Description: Headline of the story, as it will appear in the brief

  2. Date Added
     Type: date
     Description: When the row was filed to the pool (typically today)

  3. Source
     Type: text
     Description: Publication or attribution string, e.g. "Anthropic,
     Reuters, CNBC, May 28, 2026"

  4. Source Link
     Type: url
     Description: Primary source URL if available

  5. Source Date
     Type: date
     Description: When the source published the news (often different
     from Date Added)

  6. Announcement Summary
     Type: text
     Description: 3-4 sentence factual summary. Shared layer.

  7. What It Means
     Type: text
     Description: 3-4 sentence general implication for organizations.
     Shared layer.

  8. Analogy
     Type: text
     Description: 1-2 sentence "in plain terms" analogy. Shared layer.

  9. Resonance
     Type: select
     Options:
       - Headline (red)
       - Research (blue)
       - Niche (gray)

 10. Relevance
     Type: multi-select
     Options:
       - M&T Bank
       - Brown Advisory
       - Alpine Associates
       - Incserv
       - ChristianaCare
       - KPS Life
       - Luna Diabetes

 11. Lens
     Type: multi-select
     Options:
       - Risk & Regulatory
       - Vendor & Platform
       - Workforce & Operating Model
       - Capital & Deals
       - Cyber
       - Engineering Velocity
       - Clinical & Trials

 12. Status
     Type: select
     Options:
       - Candidate (default for new rows)
       - Drafted
       - Used
       - Archived

 13. Capture Source
     Type: select
     Options:
       - Daily Scan (blue)
       - Grok Email (orange)
       - Newsletter (green)
       - Manual (gray)

 14. Week
     Type: select
     Options:
       - W21, W22, W23, W24... (one per ISO calendar week)
       - Colors rotate for visual distinction

VIEWS (8 functional, 10 total including all per-client views)

  Default
    Type: Table
    Filter: none
    Sort: Date Added, descending

  This Week
    Type: Table
    Filter: Date Added is within last 7 days
    Sort: Resonance (Headline > Research > Niche), then Date Added desc

  By Week
    Type: Table, grouped
    Group by: Week
    Sort within group: Resonance, then Date Added desc
    Order of groups: manual (most recent week on top)

  KPS Life
    Type: Table
    Filter: Relevance contains KPS Life AND Status is not Archived
    Sort: Resonance, then Date Added desc

  M&T Bank, Brown Advisory, Alpine Associates, Incserv,
  ChristianaCare, Luna Diabetes
    Each: same filter pattern as KPS Life, scoped to that client's
    Relevance value

APPENDIX B: GROK SCHEDULED TASK PROMPT
================================================================================

Paste this prompt into the body of the grok.com/tasks scheduled task.
Update the date references and email address as needed.

--- BEGIN GROK PROMPT ---

You are running a daily scan for Keel3's weekly intelligence brief. Pull
posts and threads from the following six X lists in the prior 24 hours:

  1. AI Newsmakers: https://x.com/i/lists/<your_list_id_1>
  2. AI Founders: https://x.com/i/lists/<your_list_id_2>
  3. AI Investors: https://x.com/i/lists/<your_list_id_3>
  4. AI Policy: https://x.com/i/lists/<your_list_id_4>
  5. Healthcare/Science: https://x.com/i/lists/<your_list_id_5>
  6. AI Researchers/Developers: https://x.com/i/lists/<your_list_id_6>

For each high-signal item in those lists, produce a structured entry with:

  - Headline (one line)
  - Source (account, publication, or X URL)
  - 3-4 sentence summary of what was announced or reported
  - 2-3 sentence read on why it matters

Pin the date you are scanning to the calendar day this task runs. Format
all dates as "Month Day, Year" (e.g., June 3, 2026).

CRITICAL RULES:

- Deliver the FULL output, not a truncated digest. Do not summarize away
  detail. The Keel3 operator needs to see the structured items.

- Flag single-post claims explicitly: if a story comes from only one X
  post or one source, note "(single-post reporting noted)" so the operator
  knows to verify before filing.

- De-emphasize crypto and token-coin framing unless the item has direct
  enterprise relevance.

- Verify numbers and named figures. If you cannot verify, do not assert.
  Use phrasing like "reported to be" or skip the number.

- At the end of the scan, include a "Noise" section listing items that
  were loud but low substance (hot takes without new data, talent gossip,
  speculative AGI claims). The operator will skip these.

Deliver via email to <operator email address>.

--- END GROK PROMPT ---

Replace the six list IDs with the actual X list IDs for the curated
sources. To create lists in X: log in, create a new list, add accounts,
